# Source Architecture Notes

This document summarizes ownership boundaries inside `src/`.

## Directory Responsibilities

- `src/bots/`
  - Bot runtime orchestrator and module lifecycle.
  - Includes message routing (`shinxbot_meta.cpp`), network integration, and boot/runtime control.
- `src/interfaces/`
  - Stable interfaces and contracts for functions/events (`processable`, bot abstraction, dynamic loading contracts).
- `src/utils/`
  - Reusable generic helpers (I/O, JSON, logging, random, conversion, HTTP, image, QQ utils, UUID).
  - `src/utils/meta_func/` contains scoped utility classes used by multiple modules.
- `src/meta_event/`
  - Meta-level events generated by runtime itself (e.g., heartbeat), separated from user message features.
- `src/dev_tools/`
  - Developer-only debug helpers that emulate OneBot behavior.
  - Script helpers live under `tools/dev_tools/` and may keep wrappers here for compatibility.

## Classification Rules

1. If code defines extension contracts or base abstractions, keep it in `interfaces`.
2. If code drives process lifecycle or module orchestration, keep it in `bots`.
3. If code is stateless and reusable across modules, keep it in `utils`.
4. If code is synthetic protocol/meta traffic, keep it in `meta_event`.
5. Avoid putting production runtime logic in `dev_tools`.

## Notes on Moving Files

Current CMake setup uses `aux_source_directory` on top-level directories (non-recursive), so deep folder moves inside these directories should be accompanied by CMake updates.
